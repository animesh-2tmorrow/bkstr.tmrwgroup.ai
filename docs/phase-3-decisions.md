## D9 — Phase 3 cross-cutting decisions (Stream 1 first patch)

### D9.1 — Role enum on `User` (`SUBSCRIBER`, `PUBLISHER`, `ADMIN`); default `SUBSCRIBER`

**Choice:** added `enum Role { SUBSCRIBER, PUBLISHER, ADMIN }` to `prisma/schema.prisma` and a `role Role @default(SUBSCRIBER)` column on `User`. Migration `20260510130000_phase_3_role_enum` adds the column with `NOT NULL DEFAULT 'SUBSCRIBER'` so the three existing user rows backfill cleanly, plus an `UPDATE ... WHERE email = 'animesh@2tmorrow.com'` that promotes the operator identity to `ADMIN` (per ROLE-Q1). Stream 1 lands this FIRST as a blocking dependency: Stream 2 (S3 authorization scope) and Stream 3 (Stripe Customer identity) both want to read role at signin or customer-creation time.

**Reasoning:** the three-role split matches the access pattern Phase 2 already implies. `SUBSCRIBER` is the existing 1:1-with-User-model (D1.3); `PUBLISHER` is the natural complement once #39's split lands (publishers see "their books and how they're being used", subscribers see "books I have access to"); `ADMIN` is the operator role for allowlist edits, manual access grants, and any future moderation surface. Filing role on `User` rather than `Subscriber` is deliberate — role describes the human identity, while `Subscriber` describes the billing entity. A future scenario where one User belongs to multiple Subscribers (D1.3's relaxation) keeps role meaningful: the same person stays an `ADMIN` even when their billing context changes. Alternatives considered: separate `user_roles` join table (rejected — premature flexibility; three roles fit a single column); free-form string column without enum (rejected — loses Postgres-level invariants). NextAuth's `Session.user` type augmentation lives in `next-auth.d.ts` so `session.user.role` is type-safe immediately.

### D9.2 — `content_uri` vs `content` dual-storage seam (Stream 2 owns the migration)

**Choice:** the schema retains both `book_versions.content_uri TEXT NOT NULL` (Phase 1 placeholder, currently `inline://<uuid>`) and `book_versions.content TEXT` (nullable, current source of truth). Phase 3 commits to a dual-storage seam with a precedence rule: read path prefers `content_uri` if non-null AND scheme `s3://`, else falls back to inline `content`. Write path during S3 migration: write to S3 first, set `content_uri = "s3://<bucket>/<key>"`, then null out `content` in a follow-up sweep. Post-migration: drop `content` column. Stream 1's job for this seam is decision-log only — both columns already exist (Phase 1 created `content_uri NOT NULL`; Phase 2 Step 3 added `content` nullable). Stream 2 owns the read-path code change, the S3 client wiring, and the migration sweep.

**Reasoning:** the alternative (drop `content_uri`, commit inline forever; or drop `content`, commit S3 forever) was considered and rejected. Inline-forever caps book size at TOAST-friendly bounds and bloats DB backups; S3-forever requires Stream 2 to land in lockstep with the migration, no graceful fallback if S3 hiccups. The dual-storage shape lets the migration roll forward incrementally — books move book-by-book, with the read path correctly serving from whichever source is populated. The `s3://` scheme check on `content_uri` is the precedence signal because today's placeholder values are `inline://<uuid>` which trivially fail the scheme check, so the read path stays correct without backfill. Closes #45 as a forward-pointer.

### D9.3 — Stream 3 lands first webhook precedent + `webhook_events` idempotency table

**Choice:** Stream 3 (Stripe) owns the first webhook route at `src/app/api/webhooks/stripe/route.ts`, the signature verification helper, and the route-level pattern. Stream 2 (S3 SNS event notifications, if needed) follows that shape. Stream 1 owns the schema work in this patch: a single `webhook_events` table keyed on `event_id VARCHAR(128)` (Stripe's `evt_…` strings) with a `source VARCHAR(32)` discriminator + `received_at`/`processed_at`/`status VARCHAR(32)` lifecycle columns + composite index on `(source, received_at DESC)`. Migration: `20260510140000_phase_3_webhook_events`.

**Reasoning:** Stripe's at-least-once delivery model produces routine duplicate webhook calls (network retries, replay during outages). Without an idempotency layer, a `customer.subscription.created` event handled twice would create two `access_grants` rows, two Stripe customer records, or worse. The `event_id`-as-PK shape (no synthetic UUID + unique index) is cheaper and matches the natural key. Generalizing to `source` means the same table covers Stream 2's SNS event idempotency without parallel schema. The `received_at` / `processed_at` separation captures "we acknowledged delivery" vs "we successfully handled" — useful for diagnosing stuck handlers and a Stripe-dashboard-shaped operations view. `status` is `VARCHAR(32)` for the same reason `fetch_logs.status` is (#20) — values stabilize after observation, then enum-ize. Stream 3 will write a small `withIdempotency(eventId, async () => {...})` wrapper that Stream 2 reuses verbatim if/when needed.

### D9.4 — Per-service env files in `/etc/bkstr/` (mode 600 root)

**Choice:** Phase 3's secret-bearing env files follow the per-service-file pattern from Phase 2's `/etc/bkstr/oauth.env`. Stream 3 ships `/etc/bkstr/stripe.env` (containing `STRIPE_SECRET_KEY`, `STRIPE_WEBHOOK_SECRET`, optionally `STRIPE_PUBLISHABLE_KEY`); Stream 2 ships `/etc/bkstr/aws.env` (containing `AWS_REGION`, `S3_BUCKET_NAME`, etc.; instance-profile credentials remain the auth source per D2.x). All files mode 600 root:root. `scripts/start.sh` sources each file before pm2 reload, matching the existing `oauth.env` pattern from D1.7.

**Reasoning:** one big `/etc/bkstr/app.env` would conflate independently-rotated secrets, complicate file-level ACLs, and make the failure mode "Stripe key missing" indistinguishable from "AWS key missing" at the loud-warn-at-startup layer. Per-service files match how secrets actually rotate (Stripe key rotation is independent of AWS access key rotation, which is independent of OAuth key rotation). The `[ -f ... ] && source ...` pattern (D1.7) means a missing service file is a logged WARN, not fatal — protects deploys from "Stripe enabled in code but key not yet staged." Out of Stream 1's direct scope but documented for cross-Stream consistency.

### D9.5 — Stripe SDK pinning (exact versions, no caret)

**Choice:** at first Stream 3 use, the Stripe Node SDK gets pinned to an exact version (e.g. `"stripe": "18.5.0"` not `"^18.5.0"`), matching D2.4's Bedrock SDK pinning rule. Today the package.json carries `"stripe": "^22.1.1"` from Phase 1; Stream 3's first commit will replace the caret with the exact version it actually uses.

**Reasoning:** Phase 1's lesson — surface bugs in build pipelines compound — argues against silent dependency drift in security-sensitive code paths. Webhook signature verification, customer creation, and subscription lifecycle handlers all live on the hot path between user action and money movement; an unexpected behavioral shift between `npm install` runs is a bug class worth foreclosing. Exact-pin is the same posture that protects Bedrock SDK from silently bumping past Node-version compatibility (#17). Stream 3 will run `npm view stripe versions` at first install, pick the latest stable, and pin. Subsequent bumps are deliberate PRs.

### D9.6 — `access_grants` full table with source enum + lifecycle (closes #32)

**Choice:** added `enum GrantSource { MANUAL, SUBSCRIPTION, PURCHASE, SEED }` and `model AccessGrant` with columns: `id`, `subscriberId`, `bookId`, `source`, `stripeSubscriptionId VARCHAR(64)?`, `stripePaymentIntentId VARCHAR(64)?`, `grantedAt`, `expiresAt?`, `revokedAt?`, `grantedBy?` (FK to `users.id`). FKs: `subscriber_id → subscribers(id) CASCADE`, `book_id → books(id) CASCADE`, `granted_by → users(id) SET NULL`. Indexes: `UNIQUE(subscriber_id, book_id, source)` (allows MANUAL + SUBSCRIPTION rows coexisting for the same pair), `(subscriber_id, book_id)` (read-path lookup), `(stripe_subscription_id)` (Stripe webhook reverse lookups). Migration `20260510150000_phase_3_access_grants` includes a SEED-source backfill for every existing `(subscriber, book)` pair (3 × 5 = 15 rows) via `INSERT ... CROSS JOIN ... ON CONFLICT DO NOTHING`.

**Reasoning:** the rejected alternatives (thin `subscriber_books` join; polymorphic subject column) were reviewed in the Stream 1 design pass. The full table with `source` discriminator gives Stream 3's webhook handler an idempotent upsert key (`subscriber_id, book_id, source`), traces every grant back to its origin (Stripe object or operator action), and supports time-bounded trials via `expires_at`. The `SEED` source is a clean provenance marker for grandfathered Phase 2 access — operators reading the table six months from now can distinguish "this is from before per-book auth was a thing" from "this came from a Stripe subscription." Per-book authorization itself is enforced via a route-side check (`prisma.accessGrant.findFirst` in `src/app/api/agent/fetch/route.ts` between the current book lookup and the version lookup) gated by an `ENFORCE_BOOK_ACCESS` env flag — that read-path change ships in a follow-up patch with the rollout sequence flag-off → seed → flag-on with telemetry watch (ACCESS-Q2's belt-and-suspenders). Lifecycle conventions match `subscriber_api_keys` (D4.5): soft-delete via `revoked_at`, no hard delete; `expires_at` nullable means "no expiry until revoked." Returns HTTP 403 with `fetch_logs.status='access_denied'` on missing grant (ACCESS-Q3) — distinct from 401 to give operators the signal during external onboarding.

### D9.7 — Per-book one-time pricing model (`BookPrice`); subscription tiers deferred to Phase 4

**Choice:** added `model BookPrice` with columns: `id`, `bookId`, `currency CHAR(3) DEFAULT 'USD'`, `unitAmountCents INT`, `stripePriceId VARCHAR(64)?`, `createdAt`, `updatedAt`. Indexes: `UNIQUE(book_id, currency)` (one active price per book per currency), `UNIQUE(stripe_price_id)`. FK `book_id → books(id) CASCADE`. Migration `20260510160000_phase_3_book_prices`. The existing `Subscription` + `SubscriptionTier` Phase 1 placeholders are left intact, unused by Stream 3, deferred to Phase 4 if recurring billing is ever scoped.

**Reasoning:** Phase 3's marketplace MVP is unlock-per-book — a subscriber pays once for a book and gets a `PURCHASE`-source `AccessGrant`. The `BookPrice` table mirrors Stripe's Product+Price separation: one Product (Book) can have multiple Prices (eventual multi-currency, but Phase 3 is USD-only with shape future-proofing per PRICING-Q2). UPDATE-in-place when price changes; the `stripe_price_id` gets repointed to a new Stripe Price object since Stripe Prices are immutable. **BookPrice has no schema-level price history. Stripe is the system-of-record (immutable Price objects). Historical pricing reconstructable via Stripe API or `webhook_events` log.** This trade-off is intentional: Stripe owns the audit trail, our schema stays small, and any local cache of historical prices would inevitably drift from Stripe's truth without active reconciliation. The rejected alternatives — extending `Subscription` with price fields (Plan agent Option B), adding a `Plan`+`PlanItem` table (Option D) — solve recurring billing problems Phase 3 doesn't have. Phase 4 may revisit when subscription is real.
